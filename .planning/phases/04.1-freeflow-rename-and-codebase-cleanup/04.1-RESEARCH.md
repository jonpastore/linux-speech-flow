## Summary

This phase is a pure housekeeping pass with zero new functionality. The codebase was built under the name "FreeFlow" during early planning, and the actual implementation used `linux_speech_flow` as the package name but retained "FreeFlow" strings in user-visible locations (tray title, icon names, notification titles, desktop file, autostart entry). This phase closes that gap.

The scope is fully enumerable: 2 source files to edit (`app.py`, `tray.py`), 8 SVG icon files to rename, 1 documentation file to update (`REQUIREMENTS.md`), 1 config migration function to add (`config.py`), and a dead code scan across all modules. The git squash is a final step that rewrites the entire repository history to a single commit.

The key risk in this phase is the icon rename affecting the live user system: icons have already been installed to `~/.local/share/icons/hicolor/scalable/status/` under the old `freeflow-*` names, and the autostart `.desktop` file at `~/.config/autostart/freeflow.desktop` references the old icon and old filename. The rename pass must update both the source files AND handle these installed artifacts. The `install_icons()` function already copies and overwrites on every startup — so renaming source files + updating the constant names in `tray.py` will cause `install_icons()` to deploy the new names automatically on next run.

**Primary recommendation:** Three plans in sequence — (1) rename icons + update tray.py/app.py string literals, (2) config migration + REQUIREMENTS.md + dead code removal, (3) git history squash as a standalone step.

## Architecture Patterns

### Pattern 1: Icon rename + reference update must be atomic

Icon files in `src/linux_speech_flow/icons/` are tracked by `pyproject.toml` as package data via `icons/*.svg`. Renaming the SVG files changes what gets included in the package. The `ICON_NAMES` list in `tray.py` is the authoritative list of what `install_icons()` copies. After rename:
1. SVG files use new names
2. `ICON_NAMES` list uses new names
3. `install_icons()` copies new names to `~/.local/share/icons/hicolor/scalable/status/`
4. Old `freeflow-*.svg` files at the installed location become orphans — must be cleaned up explicitly

### Pattern 2: Autostart desktop file cleanup

The `_install_autostart()` method in `app.py` writes the desktop file on every `do_startup()` call. After the rename:
- New code writes to `~/.config/autostart/linux-speech-flow.desktop`
- Old file `~/.config/autostart/freeflow.desktop` persists as an orphan and will cause double-autostart on next login

Two options:
1. Delete the old desktop file in `_install_autostart()` before writing the new one (one-time migration within the function)
2. Rely on the user to manually remove it

The right approach (per P1_INTEGRITY) is to remove the old autostart file as part of `_install_autostart()`.

### Pattern 3: Config migration in config.py (before load)

The migration must run before any `load_config()` call. The cleanest location is a `migrate_config_paths()` function called at module import time or at the top of `load_config()`. Per CONTEXT.md, `config.py` is the right place.

```python
# Pattern: add to config.py

OLD_CONFIG_DIR = Path.home() / ".config" / "freeflow"

def _migrate_legacy_config() -> None:
    if not OLD_CONFIG_DIR.exists():
        return
    if CONFIG_PATH.exists():
        # New config already exists; skip migration, just clean up old dir
        shutil.rmtree(OLD_CONFIG_DIR, ignore_errors=True)
        return
    # Migrate: copy old config to new location
    old_config = OLD_CONFIG_DIR / "config.json"
    if old_config.exists():
        CONFIG_DIR.mkdir(parents=True, exist_ok=True)
        shutil.copy2(old_config, CONFIG_PATH)
    shutil.rmtree(OLD_CONFIG_DIR, ignore_errors=True)
```

Call from `load_config()` at the top, before file read. Use logging to report migration. Silent on success, warning on failure (per CONTEXT.md specifics).

### Pattern 4: Git history squash

Current state: 88 commits on `main`. The squash must produce a single commit with message "Initial commit: Linux Speech Flow".

Standard approach using `git reset` (not interactive rebase, which is interactive-terminal-dependent):

```bash
# Get the root commit hash
ROOT=$(git rev-list --max-parents=0 HEAD)
# Soft reset to root, keeping all changes staged
git reset --soft $ROOT
# Amend the root commit with new message
git commit --amend -m "Initial commit: Linux Speech Flow"
```

Alternative with orphan branch:
```bash
git checkout --orphan fresh-start
git add -A
git commit -m "Initial commit: Linux Speech Flow"
git branch -D main
git branch -m main
git push --force origin main  # only if remote exists
```

**Critical:** The squash is destructive and irreversible. It must be the final step after all other changes are committed and verified.

## Common Pitfalls

### Pitfall 1: Old installed icons not removed

**What goes wrong:** After renaming source SVGs and updating `ICON_NAMES`, `install_icons()` copies new `linux-speech-flow-*.svg` files. But old `freeflow-*.svg` files remain at `~/.local/share/icons/hicolor/scalable/status/`. The tray will work (new icons found), but the filesystem has orphaned files.

**How to avoid:** Add cleanup of old icon names in `install_icons()` or as a one-time migration step:
```python
for old_name in ['freeflow-idle', 'freeflow-error', ...]:
    old_path = dest_dir / f'{old_name}.svg'
    old_path.unlink(missing_ok=True)
```

**Warning signs:** Both old and new SVG files in `~/.local/share/icons/hicolor/scalable/status/`.

### Pitfall 2: Double autostart entry

**What goes wrong:** After the rename, `_install_autostart()` creates `linux-speech-flow.desktop`. The old `freeflow.desktop` remains in `~/.config/autostart/`. On next login, both entries fire, launching two instances. The Gtk.Application D-Bus single-instance enforcement prevents a second window but creates a confusing startup error.

**How to avoid:** In `_install_autostart()`, explicitly unlink the old desktop file before writing the new one:
```python
old_desktop = autostart_dir / 'freeflow.desktop'
old_desktop.unlink(missing_ok=True)
```
**Confirmed:** `~/.config/autostart/freeflow.desktop` EXISTS on this machine. This cleanup is required.

### Pitfall 3: Icon name mismatch breaks tray entirely

**What goes wrong:** If SVG files are renamed but `ICON_NAMES` in `tray.py` is not updated (or vice versa), `install_icons()` either copies files under old names or tries to copy non-existent files. The tray icon will fail to display and may throw an exception.

**How to avoid:** The icon rename and `tray.py` constant update must be in the same plan/commit. Never half-rename.

### Pitfall 4: Config migration runs after load_config is called

**What goes wrong:** If `_migrate_legacy_config()` is called inside `app.py`'s `do_startup()` rather than inside `config.py`'s `load_config()`, there's a risk of `load_config()` being called elsewhere before migration runs (e.g., in a test or from another module).

**How to avoid:** Place the migration call inside `load_config()` itself, at the very top, before the `_path.exists()` check. This ensures migration always happens before any config read, regardless of call site.

### Pitfall 5: Git squash without verifying working tree is clean

**What goes wrong:** If there are uncommitted changes when the squash runs, `git reset --soft` may include them in the single commit or leave them in a confusing state.

**How to avoid:** Run `git status` to confirm clean working tree before squash. Make the squash the final step.

### Pitfall 6: pyproject.toml package-data glob catches renamed files

**What goes wrong:** `pyproject.toml` includes `"icons/*.svg"` as package data. After renaming, the glob still matches both old and new filenames if the old files weren't removed from the git index. The build will include both sets.

**How to avoid:** Use `git mv` to rename SVG files. This stages the rename correctly. After `git mv`, the old filename is gone from git's perspective.
