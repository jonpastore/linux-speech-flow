    In `TranscriptionPipeline.__init__`, add `on_failed_count_changed=None` as a parameter. Store as `self._on_failed_count_changed = on_failed_count_changed`.

    Add two private helpers immediately after `__init__`:
    ```python
    def _get_failed_count(self) -> int:
        return len(list(FAILED_DIR.glob('*.wav'))) if FAILED_DIR.exists() else 0

    def _notify_failed_count(self):
        if self._on_failed_count_changed:
            count = self._get_failed_count()
            GLib.idle_add(self._on_failed_count_changed, count)
    ```

    Call `self._notify_failed_count()` in two places:
    1. At the end of `_save_failed_wav()` — after the `shutil.move()` succeeds (inside the try block, after the `logger.info` line). This handles count increases.
    2. In `_process()`: after `os.unlink(wav_path)` in the batch path section (line ~251 in current source), and after `os.unlink(wav_path)` in the normal paste path (line ~261). Both `os.unlink` calls are in try blocks; call `self._notify_failed_count()` after the try/except in each location. This handles reprocess count decreases when WAVs from failed/ are processed (wav_path starts with str(FAILED_DIR) is implicitly handled — any unlink of a WAV decreases count via fresh glob, no path-checking needed since the failed count is always from directory state).

    NOTE: `update_failed_count` in TrayManager returns `False` (as coded in plan 01). When dispatched via `GLib.idle_add(self._on_failed_count_changed, count)`, GLib.idle_add passes `count` as the argument and the callback returns `False`, which is correct — idle_add removes the callback after it returns False (single fire). This matches the established project pattern.
