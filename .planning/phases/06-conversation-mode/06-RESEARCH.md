## Summary

Phase 6 adds a second recording mode ("Conversation Mode") that runs alongside but completely separate from the existing push-to-talk (F9) transcription pipeline. The core technical challenge is a state machine that manages: a long-running chunked recorder, background per-chunk transcription, multiple silence timers, a post-stop analysis dialog, and an iterative Q&A loop — all while preserving the existing pipeline untouched.

The project already has all foundational pieces: `AudioRecorder` for WAV capture, `TranscriptionPipeline` for Groq Whisper + LLM, `HotkeyManager` for pynput key dispatch, and `HistoryStore`/SQLite for persistence. Phase 6 adds a `ConversationRecorder` (chunked, long-running variant), a `ConversationPipeline` (multi-model AI analysis), three new GTK windows (status, post-stop dialog, Q&A), and a conversation file viewer. The existing codebase must not be modified except at well-defined extension points.

Multi-model support requires two new pip dependencies: `google-genai` (Gemini, replaces deprecated `google-generativeai`) and `openai` (for Grok via OpenAI-compatible interface at `https://api.x.ai/v1`). Both are thin clients that follow the same request/response shape as the existing Groq SDK — the pattern transfers directly.

**Primary recommendation:** Implement ConversationRecorder as a new class (not a subclass of AudioRecorder) that writes silence-chunked WAV segments to a temp directory, with a background thread per chunk dispatching Whisper transcription. Keep all conversation state in a dedicated ConversationManager class owned by App, mirroring how TranscriptionPipeline is owned today.

### Pattern 2: GLib Timer for Silence Thresholds
**What:** Three independent GLib.timeout_add_seconds timers manage conversation-level silence (180s warn, 300s auto-stop, 14400s hard limit). These are distinct from the per-chunk silence detection in ConversationRecorder which uses RMS.

```python
# Source: GLib docs + project pattern in hotkey.py / transcription.py
from gi.repository import GLib

class ConversationManager:
    def _start_silence_timers(self):
        # Called when last chunk finished (no new speech detected at session level)
        self._warn_timer = GLib.timeout_add_seconds(
            180, self._on_silence_warn
        )

    def _reset_silence_timers(self):
        # Called when new chunk starts (speech resumed)
        if self._warn_timer:
            GLib.source_remove(self._warn_timer)
            self._warn_timer = None
        if self._stop_timer:
            GLib.source_remove(self._stop_timer)
            self._stop_timer = None

    def _on_silence_warn(self) -> bool:
        # Show GTK modal "Continue or stop?" — returns False (timer fires once)
        self._show_silence_modal()
        self._stop_timer = GLib.timeout_add_seconds(120, self._on_silence_autostop)
        return False

    def _on_silence_autostop(self) -> bool:
        play_sound("stop.wav", ...)
        self._stop_conversation()
        return False
```

**Note:** Session-level silence timers operate on `on_chunk_ready` callbacks — when a chunk arrives, silence timers reset. Timers only start when the recorder is running but no new chunk has arrived for N seconds.

### Pattern 4: Multi-Model AI Analysis
**What:** Fire analysis calls to all selected models in parallel threads, collect results, then call synthesis (meta-model) with all responses to produce unified output. Use Groq JSON mode for confidence signaling.

```python
# Source: groq SDK + google-genai SDK + openai SDK (verified with Context7)
import concurrent.futures, json

def _analyze_all_models(self, transcript: str, prompt: str, models: list[str]) -> dict:
    """Returns {"summary": str, "confidence": float, "questions": list[str]}"""
    results = {}
    with concurrent.futures.ThreadPoolExecutor() as pool:
        futures = {pool.submit(self._call_model, m, transcript, prompt): m for m in models}
        for future, model_name in futures.items():
            try:
                results[model_name] = future.result(timeout=60)
            except Exception as exc:
                results[model_name] = f"[Error: {exc}]"
    return self._synthesize(results)

def _call_groq(self, transcript, prompt) -> str:
    response = self._groq_client.chat.completions.create(
        model=self._config["conv_groq_model"],
        messages=[{"role": "user", "content": f"{prompt}\n\n{transcript}"}],
        response_format={"type": "json_object"},
        temperature=0.3,
        max_tokens=4096,
    )
    return response.choices[0].message.content

def _call_gemini(self, transcript, prompt) -> str:
    from google import genai as google_genai
    from google.genai import types
    client = google_genai.Client(api_key=self._config["gemini_api_key"])
    response = client.models.generate_content(
        model="gemini-2.5-flash",
        contents=f"{prompt}\n\n{transcript}",
        config=types.GenerateContentConfig(
            response_mime_type="application/json",
            temperature=0.3,
            max_output_tokens=4096,
        ),
    )
    return response.text

def _call_grok(self, transcript, prompt) -> str:
    from openai import OpenAI
    client = OpenAI(
        api_key=self._config["grok_api_key"],
        base_url="https://api.x.ai/v1",
    )
    response = client.chat.completions.create(
        model="grok-3-mini",
        messages=[
            {"role": "system", "content": "Return JSON only."},
            {"role": "user", "content": f"{prompt}\n\n{transcript}"},
        ],
        response_format={"type": "json_object"},
        temperature=0.3,
        max_tokens=4096,
    )
    return response.choices[0].message.content
```

### Anti-Patterns to Avoid
- **Modifying AudioRecorder**: ConversationRecorder must be a separate class. AudioRecorder's single-file model is wrong for chunked long recording.
- **Using Gtk.Dialog**: Deprecated since GTK 4.10. Use Gtk.Window(modal=True) per project pattern.
- **Blocking GTK main thread**: All AI API calls run in worker threads; results dispatched via GLib.idle_add.
- **Storing chunks in memory**: Write each silence-bounded chunk to disk immediately; 4-hour recordings at 16kHz mono S16LE = ~450MB — memory buffers are unsafe.
- **Sharing state between conversation and regular pipelines**: Keep ConversationManager completely separate from TranscriptionPipeline.
- **GTK3 Paned API (pack1/pack2)**: GTK4 uses `set_start_child()`/`set_end_child()`.
- **Using google-generativeai**: Reached EOL Nov 2025. Use `google-genai` (`from google import genai`).
- **Using openai 0.x**: Project venv has 0.27.5 but it's incompatible with 1.x API. pyproject.toml must pin `openai>=1.0.0`.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Silence-bounded WAV chunk splitting | Custom FFmpeg subprocess chunker | RMS-based inline split in ConversationRecorder using existing recorder.py logic | Already proven in recorder.py; pasimple gives raw frames; FFmpeg adds process overhead |

## Common Pitfalls

### Pitfall 1: Whisper API File Size Limit
**What goes wrong:** Groq Whisper accepts up to 100MB on paid tier (25MB free tier). A 4-hour WAV at 16kHz mono S16LE = ~460MB per full recording. If the whole session WAV is sent in one call, it will be rejected.
**Why it happens:** Developer assumes the file size limit applies to the final output, not the upload.
**How to avoid:** ConversationRecorder MUST deliver silence-bounded WAV chunks in real-time (3–10 minutes of speech each). Each chunk is transcribed independently. The full session WAV is never assembled — only the final coalesced text.
**Warning signs:** Any design where transcription happens "after" recording stops instead of in parallel.

### Pitfall 2: Conversation State Leaking into Regular Recording
**What goes wrong:** Fn+C pressed during a conversation starts regular transcription instead of or alongside conversation mode.
**Why it happens:** HotkeyManager has only one `_state` variable shared between all recording modes.
**How to avoid:** Add a `_STATE_CONVERSATION` to HotkeyManager that blocks F9 recording while conversation mode is active (and vice versa). The simplest approach: if `_state != _STATE_IDLE`, ignore the competing key.
**Warning signs:** Fn+C during Fn+9 recording does something unexpected.

### Pitfall 3: GTK Main Thread Violations in Q&A Loop
**What goes wrong:** Q&A loop calls API directly from a button-click handler, blocking the GTK main thread. UI freezes during API calls (can be 5–30 seconds).
**Why it happens:** The async pattern is non-obvious; blocking calls in GTK handlers feel "simpler."
**How to avoid:** The Q&A submit handler always dispatches to a daemon thread (same pattern as TranscriptionPipeline). Result is delivered via GLib.idle_add to update the Q&A window.
**Warning signs:** "Submit" button click causes window to become unresponsive.

### Pitfall 4: openai 0.x vs 1.x API Incompatibility
**What goes wrong:** The project venv has `openai==0.27.5` installed. Code written with 1.x `openai.OpenAI()` constructor raises `AttributeError` at runtime.
**Why it happens:** 0.x used module-level functions (`openai.ChatCompletion.create()`); 1.x uses a client instance (`client.chat.completions.create()`).
**How to avoid:** Add `openai>=1.0.0` to pyproject.toml dependencies explicitly. The pip install will upgrade the existing 0.27.5.
**Warning signs:** `AttributeError: module 'openai' has no attribute 'OpenAI'`

### Pitfall 5: Session-Level vs Chunk-Level Silence Detection
**What goes wrong:** The 180s/300s timers fire based on the wrong silence signal. Two types of silence exist: (a) brief natural pauses between utterances (handled by ConversationRecorder to bound WAV chunks), (b) extended session-level silence indicating the user has stopped speaking (handled by ConversationManager timers).
**Why it happens:** Conflating the two silence concepts into one counter.
**How to avoid:** ConversationRecorder emits `on_chunk_ready` (bounded silence found, new chunk started). ConversationManager resets session timers on each `on_chunk_ready`. Session timers only fire if no new chunk has arrived for 180s.
**Warning signs:** Silence timer fires after every pause in speech; or never fires on genuine session silence.

### Pitfall 6: GTK4 Paned API Mismatch
**What goes wrong:** Code uses `paned.pack1(child, resize=True, shrink=False)` which is GTK3 API. In GTK4 this raises `AttributeError`.
**Why it happens:** Online examples and tutorials often reference GTK3.
**How to avoid:** Use `paned.set_start_child(child)` and `paned.set_end_child(child)` which are the GTK4 methods.
**Warning signs:** `AttributeError: 'Paned' object has no attribute 'pack1'`

### Pitfall 7: File Rename Race Condition
**What goes wrong:** `ISO8601_untitled.txt` is renamed to `ISO8601_ai-title.txt` while the viewer is open, breaking the viewer's file path reference.
**Why it happens:** The viewer holds the original path; rename invalidates it silently.
**How to avoid:** The conversation viewer must store the directory, not the full path. On any file-system event (or on re-open), it re-scans the directory. Alternatively: viewer always re-reads by listing `~/Documents/conversations/*.txt` sorted by mtime.

### Pitfall 8: Groq JSON Mode Requires Explicit Schema in Prompt
**What goes wrong:** `response_format={"type": "json_object"}` returns valid JSON but the schema varies between calls (sometimes `confidence_level`, sometimes `confidence`, etc.).
**Why it happens:** JSON mode forces JSON output but doesn't enforce schema without explicit instructions in the system prompt.
**How to avoid:** Include the exact schema with field names and types in the system prompt. Parse defensively: `parsed.get("confidence", 0.0)` with a float default.

### GLib Elapsed Timer for Status Window
```python
# Source: GLib docs (docs.gtk.org/glib/func.timeout_add.html) + project pattern
import time

class ConversationStatusWindow(Gtk.ApplicationWindow):
    def _start_timer(self):
        self._started_at = time.monotonic()
        self._timer_id = GLib.timeout_add_seconds(1, self._update_elapsed)

    def _update_elapsed(self) -> bool:
        elapsed = int(time.monotonic() - self._started_at)
        h, rem = divmod(elapsed, 3600)
        m, s = divmod(rem, 60)
        self._elapsed_label.set_text(f"{h:01d}:{m:02d}:{s:02d}")
        return True  # keep firing

    def _stop_timer(self):
        if self._timer_id:
            GLib.source_remove(self._timer_id)
            self._timer_id = None
```

from datetime import datetime

def conv_filename(title: str = "untitled") -> str:
    ts = datetime.now().strftime("%Y%m%dT%H%M%S")
    safe_title = "".join(c if c.isalnum() or c in "-_ " else "" for c in title)

### HotkeyManager Extension for Fn+C and Fn+D
```python
# Source: hotkey.py pattern — extend _on_press
_STATE_IDLE = "idle"
_STATE_RECORDING = "recording"
_STATE_CONVERSATION = "conversation"  # new

def _on_press(self, key):
    if not self._started:
        return
    # Existing F9/ESC logic (unchanged)
    if key == keyboard.Key.f9 and self._state == self._STATE_IDLE:
        GLib.idle_add(self._start_recording)
    elif key == keyboard.Key.f9 and self._state == self._STATE_RECORDING:
        GLib.idle_add(self._stop_recording_f9)
    elif key == keyboard.Key.esc and self._state == self._STATE_RECORDING:
        GLib.idle_add(self._stop_recording, False)
    elif key == keyboard.Key.f10:
        GLib.idle_add(self._on_f10)
    # New conversation hotkeys
    elif key == keyboard.Key.f11 and self._state == self._STATE_IDLE:
        GLib.idle_add(self._on_conv_start)
    elif key == keyboard.Key.f11 and self._state == self._STATE_CONVERSATION:
        GLib.idle_add(self._on_conv_stop)
    elif key == keyboard.Key.f12 and self._state == self._STATE_CONVERSATION:
        GLib.idle_add(self._on_conv_toggle_feedback)
```
**Note on key mapping:** "Fn+C" is a laptop function key combo that maps to a specific Function key depending on the keyboard layout. The CONTEXT.md says "Fn+C is the Phase 6 default" for start/stop, and "Fn+D" for feedback toggle. On most Linux keyboards, `Fn+C` does not produce a standard pynput Key. Phase 7 will make these configurable; for Phase 6, use `keyboard.Key.f11` for conversation start/stop and `keyboard.Key.f12` for feedback toggle as named placeholders that work across hardware. Document the actual key in the config so Phase 7 can swap it.

## Open Questions

1. **What pynput key does Fn+C actually generate on the user's hardware?**
   - What we know: Fn+C produces a hardware-dependent keycode; on most Linux keyboards it may generate Key.f3 or Key.f11 or nothing at pynput level
   - What's unclear: Without access to the target hardware to test, the exact pynput Key enum value is unknown
   - Recommendation: Use keyboard.Key.f11 for conversation start/stop as the Phase 6 default. Store the key name in config (`conv_hotkey_start: "f11"`) so Phase 7 can make it configurable. Document this in the plan task explicitly.

2. **Which Grok model to use as default?**
   - What we know: xAI has grok-3, grok-3-mini, grok-2-latest; all accessible at `https://api.x.ai/v1`
   - What's unclear: Current pricing, context window size — research at implementation time from xAI docs
   - Recommendation: Default to `grok-3-mini` (lower cost, faster) with user-configurable model string in config.

3. **Does the conversation entry need to be in the existing history DB or a separate file system?**
   - What we know: The history DB has `entry_type` + `extra_json` columns added in Phase 5 specifically for Phase 6 extensibility; the CONTEXT.md specifies a flat file system (`~/Documents/conversations/`)
   - What's unclear: The CONTEXT.md describes a dedicated conversation file viewer (not the history window), implying flat files are the primary storage
   - Recommendation: Use flat `.txt` files as primary storage (as per CONTEXT.md) for conversation content. Optionally insert a lightweight `entry_type='conversation'` row in history DB with just metadata (date, duration, ai_title, file_path in extra_json) so it can appear in the history window if desired in a future phase. Phase 6 plan should not add history DB tracking unless explicitly scoped.

4. **How should the status window be positioned?**
   - What we know: Claude's Discretion; no position specified
   - What's unclear: Whether it should be persistent, or appear/dismiss on Fn+D
   - Recommendation: Status window spawns when conversation starts if feedback_mode == "status_window"; closes when conversation stops. Fn+D toggles between tray-only (closes window if open) and status_window (opens window if closed).
