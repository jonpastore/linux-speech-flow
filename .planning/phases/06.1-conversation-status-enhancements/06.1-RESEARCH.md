## Summary

This phase is a focused UI-and-data-plumbing task. All the hard work already exists: `ConversationRecorder._record_one_chunk` counts `silence_frames` at every 100ms audio chunk; `ConversationManager._on_chunk_transcribed` already receives the transcript text. Neither class surfaces these values to `ConversationStatusWindow`. The work is: (1) emit silence state from recorder to manager via a new callback, (2) track elapsed silence seconds in manager, (3) forward both silence_seconds and last_transcript to the status window, and (4) add two new UI rows to the status window.

The threading model is already established and must be respected: the recording loop runs in a daemon thread and must only touch GTK indirectly via `GLib.idle_add`. The silence tick callback is the minimal new cross-thread signal needed. The manager already uses `GLib.idle_add` for all its GTK interactions, so the pattern is proven.

Layout discretion belongs to Claude. The recommended approach: keep the existing elapsed label (title-1) and status label; add a "Silence: Xs" label (prominently sized but smaller than title-1); add a wrapping last-transcript label (smaller, muted). Increase window height from 120px to ~200px to fit comfortably.

**Primary recommendation:** Add an `on_silence_tick` callback to `ConversationRecorder.start()`, emit it each audio frame loop iteration from `_record_one_chunk`, plumb it through `ConversationManager` to `ConversationStatusWindow` via `GLib.idle_add` — then add two new `Gtk.Label` rows to the status window.

## Architecture Patterns

### Recommended Project Structure

No new files are needed. Changes are confined to:

```
src/linux_speech_flow/
├── conversation_recorder.py    # Add on_silence_tick callback support
├── conversation_manager.py     # Wire silence tick to status window
└── conversation_status.py      # Add silence_seconds label + last_transcript label
```

### Pattern 1: Callback Injection for Cross-Thread Signals

**What:** `ConversationRecorder.start()` already accepts `on_chunk_ready` and `on_error` callbacks. Add `on_silence_tick(is_silent: bool, silence_seconds: int)` as an optional third callback. The recording loop emits it each iteration with the current state.

**When to use:** Any time the recorder thread needs to push state to the GTK thread without polling.

**Example (recorder side):**
```python
# In ConversationRecorder.__init__:
self._on_silence_tick = None

# In ConversationRecorder.start():
def start(self, on_chunk_ready, on_error, on_silence_tick=None) -> None:
    self._on_chunk_ready = on_chunk_ready
    self._on_error = on_error
    self._on_silence_tick = on_silence_tick
    ...

# In ConversationRecorder._record_one_chunk(), after the is_silent determination:
if self._on_silence_tick:
    GLib.idle_add(self._on_silence_tick, rms < SILENCE_RMS_THRESHOLD, silence_frames)
```

**Note:** `silence_frames` is a frame count (each frame = 0.1 seconds = CHUNK_DURATION). Convert to seconds with `int(silence_frames * CHUNK_DURATION)` on either side. Emitting the raw frame count avoids floating-point drift.

### Pattern 2: Manager-Side Silence Seconds Forwarding

**What:** `ConversationManager._on_silence_tick` (the callback registered with the recorder) receives `is_silent` and `silence_frames`. It converts to `silence_seconds` and calls `self._status_window.update_silence(silence_seconds)` on the GTK main thread. Because `GLib.idle_add` from the recorder already dispatches to the GTK main thread, no additional idle_add wrapper is needed inside the manager callback.

**Example:**
```python
# In ConversationManager.start_session():
self._recorder.start(
    on_chunk_ready=self._on_chunk_ready,
    on_error=self._on_recorder_error,
    on_silence_tick=self._on_silence_tick,
)

# In ConversationManager:
def _on_silence_tick(self, is_silent: bool, silence_frames: int) -> bool:
    if self._status_window:
        silence_sec = int(silence_frames * 0.1)  # CHUNK_DURATION = 0.1
        self._status_window.update_silence(silence_sec)
    return False  # GLib.idle_add must return False to avoid re-invocation
```

### Pattern 3: Last Transcript Update (Already Half-Wired)

**What:** `ConversationManager._on_chunk_transcribed` already has `text`. The current call to `update_status()` passes `"last chunk: just now"` — simply also pass `text` to a new `update_transcript(text)` method or extend `update_status()`.

**Recommendation:** Add a separate `update_transcript(text: str)` method to `ConversationStatusWindow` to keep the silence-seconds and transcript concerns independent. This avoids bloating `update_status()`'s signature.

**Example:**
```python
# In ConversationManager._on_chunk_transcribed:
if self._status_window and text:
    self._status_window.update_transcript(text)
```

### Pattern 4: ConversationStatusWindow New Rows

**What:** Add two `Gtk.Label` widgets to the vertical box. The status window's `__init__` builds the box; append after the existing `_status_label`.

**Recommended layout order (top to bottom):**
1. `_elapsed_label` (existing, title-1 — large clock)
2. `_status_label` (existing, "N chunks transcribed | ...")
3. `_silence_label` (new, "Silence: 0s" — monospace or body-1, prominent)
4. `_transcript_label` (new, wrapping, smaller — shows last chunk words)

**Recommended window height:** 200px (from 120px). The transcript label will wrap at 2–3 lines comfortably.

**Example:**
```python
# In ConversationStatusWindow.__init__:
self.set_default_size(360, 200)

self._silence_label = Gtk.Label(label="Silence: 0s")
self._silence_label.set_halign(Gtk.Align.START)
box.append(self._silence_label)

self._transcript_label = Gtk.Label(label="")
self._transcript_label.set_wrap(True)
self._transcript_label.set_wrap_mode(Pango.WrapMode.WORD_CHAR)
self._transcript_label.set_halign(Gtk.Align.START)
self._transcript_label.set_max_width_chars(48)
box.append(self._transcript_label)

# New methods:
def update_silence(self, silence_seconds: int) -> None:
    self._silence_label.set_text(f"Silence: {silence_seconds}s")

def update_transcript(self, text: str) -> None:
    self._transcript_label.set_text(text)
```

**Note on Pango import:** `Pango.WrapMode.WORD_CHAR` requires `from gi.repository import Pango`. If wrap mode is not critical, `set_wrap(True)` alone is sufficient (uses WORD by default).

### Anti-Patterns to Avoid

- **Touching GTK from the recorder thread directly:** The recorder runs in a daemon thread. All GTK label updates must go through `GLib.idle_add`. The `on_silence_tick` callback is dispatched via `GLib.idle_add` from within the recorder loop — do not call GTK label methods directly in `_record_one_chunk`.
- **Emitting silence tick too frequently:** Emitting once per audio frame (every 0.1s = 10 Hz) is fine for UI purposes but generates 10 idle_add calls per second. Alternative: only emit when silence_seconds changes (debounce to 1 Hz). This is preferred to avoid GTK queue pressure.
- **Using a GLib.timeout_add_seconds in the manager to count silence:** This would drift from the actual recorder's silence detection and could show incorrect values. Use the recorder's own silence_frames count as the authoritative source.
- **Modifying update_status() signature to add more positional args:** The method is already called with `(chunk_count, last_status)` in two places in the manager. Adding more positional args risks call-site breakage. Use separate methods.

|---------|-------------|-------------|-----|

**Key insight:** The recorder already tracks silence frames with sub-100ms precision. Surface that count — don't rebuild it.

## Common Pitfalls

### Pitfall 1: GLib.idle_add Callback Must Return False

**What goes wrong:** If an `idle_add` callback returns `True`, GLib re-queues it indefinitely, causing an infinite loop of label updates.
**Why it happens:** GLib.idle_add uses the return value as a "keep running" signal, same as timeout callbacks.
**How to avoid:** All `_on_silence_tick`, `update_silence`, and `update_transcript` callbacks must `return False`.
**Warning signs:** CPU usage climbs steadily after recording starts; status window flickers.

### Pitfall 2: Silence Tick Emission Rate Too High

**What goes wrong:** Emitting `on_silence_tick` every 100ms (10 Hz) floods the GLib main loop with idle_add calls, potentially delaying other GTK events.
**Why it happens:** The recorder loop runs at 100ms per iteration (CHUNK_DURATION = 0.1).
**How to avoid:** Only emit the silence tick when `silence_frames` changes value (i.e., increment or reset). This means emitting at most once per 100ms on the first change, and then again only on state transitions.
**Better pattern:** Track `_last_emitted_silence_frames` in recorder and only call idle_add when the value changes.

### Pitfall 3: Recorder Callback Called After Status Window Destroyed

**What goes wrong:** If stop_session() closes the status window while an idle_add tick is still pending, the callback fires after `self._status_window = None`. Calling methods on `None` raises `AttributeError`.
**Why it happens:** GLib.idle_add queues callbacks; there's a small window between `stop()` and the window being set to None.
**How to avoid:** Always guard: `if self._status_window:` before calling any window method. This is already the pattern in `_on_chunk_ready` and `_on_chunk_transcribed`.

### Pitfall 4: Silence Frames Count Misinterpretation

**What goes wrong:** Displaying `silence_frames` directly (e.g., "30 frames") instead of converting to seconds.
**Why it happens:** `silence_frames` is the raw count from `_record_one_chunk`, where each frame = CHUNK_DURATION = 0.1 seconds.
**How to avoid:** Convert: `silence_seconds = int(silence_frames * CHUNK_DURATION)` or equivalently `int(silence_frames * 0.1)`.

### Pitfall 5: Silence Frames Counts Silence WITHIN a Chunk, Resets Each Chunk

**What goes wrong:** Expecting `silence_frames` to be a continuous session-level counter. It is NOT — it resets to 0 at the start of each new chunk in `_record_one_chunk`.
**Why it happens:** `_record_one_chunk` is called once per chunk. Silence accumulates within a chunk until it hits `silence_limit_frames` (which ends the chunk). Then a new chunk starts and silence_frames resets.
**How to avoid:** The `on_silence_tick` should emit the CURRENT silence_frames value from within the chunk loop. When a new chunk starts, it will reset to 0 — which is correct behavior (voice was detected, silence timer resets to 0).

### Pitfall 6: is_silent vs silence_frames Ambiguity

**What goes wrong:** Checking `is_silent` alone doesn't tell you how many seconds of silence have elapsed.
**Why it happens:** A frame can be silent (rms < threshold) but only 0.1 seconds in.
**How to avoid:** Emit `silence_frames` (the cumulative count within the current chunk) as the authoritative counter. Derive `is_silent` from `silence_frames > 0` if needed, or emit both.

## Code Examples

### Full Silence Tick Plumbing (Verified Pattern — Derived from Existing Codebase)

**In `conversation_recorder.py` — extend start() and _record_one_chunk():**

```python
# _record_one_chunk: emit tick after each frame (with debounce)
def _record_one_chunk(self, pa, wav_path: str, silence_limit_frames: int):
    silence_frames = 0
    frames_written = 0
    had_audio = False
    last_emitted_silence = -1  # debounce: only emit when value changes

    with wave.open(wav_path, "wb") as wf:
        wf.setnchannels(CHANNELS)
        wf.setsampwidth(SAMPLE_WIDTH)
        wf.setframerate(SAMPLE_RATE)

        while not self._stop_event.is_set():
            raw = pa.read(CHUNK_BYTES)
            wf.writeframes(raw)
            frames_written += 1

            if frames_written >= MIN_GUARD_FRAMES:
                samples = struct.unpack(f"{len(raw) // SAMPLE_WIDTH}h", raw)
                rms = math.sqrt(sum(s * s for s in samples) / len(samples)) / 32768.0
                if rms < SILENCE_RMS_THRESHOLD:
                    silence_frames += 1
                else:
                    silence_frames = 0
                    had_audio = True

                # Emit tick only when silence_frames changes (debounced)
                if self._on_silence_tick and silence_frames != last_emitted_silence:
                    last_emitted_silence = silence_frames
                    GLib.idle_add(self._on_silence_tick, silence_frames)

                if silence_frames >= silence_limit_frames:
                    break

    return frames_written, had_audio
```

## Open Questions

1. **Silence frames reset on new chunk — UI shows 0s briefly between chunks**
   - What we know: When a silence boundary ends a chunk, `_record_one_chunk` returns and immediately starts a new chunk. In the new chunk, `silence_frames` starts at 0 (voice detected = 0 silence seconds).
   - What's unclear: Will this feel correct to the user? The chunk boundary IS a voice-detection event (the recorder's silence limit triggered), so resetting to 0 is semantically wrong — the silence IS the trigger.
   - Recommendation: Emit the tick with `silence_frames = 0` when a new chunk starts (which is already what happens when `_record_one_chunk` begins). The user will see the timer reset to 0 whenever a new chunk begins transcription, which correlates with voice being detected. Acceptable UX.

2. **Silence timer during the guard frames window (first 10 frames = 1 second)**
   - What we know: `MIN_GUARD_FRAMES = 10`. During this window, silence detection is skipped, so `on_silence_tick` won't fire for the first ~1 second of each chunk.
   - What's unclear: Should the UI show "Silence: 0s" or no update during guard frames?
   - Recommendation: Don't emit tick during guard frames (existing behavior — just don't add the idle_add call inside the guard). The label stays at whatever its last value was. Acceptable.

3. **Transcript label height — wrapping vs truncation**
   - What we know: Window will grow to ~200px. A 48-char label wrapping at 2 lines fits within that.
   - What's unclear: Very long transcript chunks (>3 lines) would overflow.
   - Recommendation: Use `set_max_width_chars(48)` with `set_wrap(True)` and no explicit line limit. Let GTK wrap naturally. The window is not resizable, so extra text will be clipped by the window boundary — acceptable for a monitoring display.
